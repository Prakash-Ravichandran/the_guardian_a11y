Role	Applicable Attributes (and related rules)														
alert	- aria-atomic (state): Indicates whether assistive technologies will present all or only parts of the changed region based on the change notifications defined by the aria-relevant attribute. Should typically be set to true. &lt;br> - aria-live (state): Indicates that an element will be updated, and describes the types of updates the user agents, assistive technologies, and user can expect from the element. Should be set to assertive.														
alertdialog	- Inherits all rules and expected attributes of dialog. &lt;br> - Typically, focus is set to an element within the alertdialog upon appearance. &lt;br> - Should clearly convey important and usually time-sensitive information to the user.														
application	- Use sparingly and only when the content truly behaves like a desktop application, not a website. &lt;br> - Requires careful management of focus and keyboard navigation. &lt;br> - Consider using more specific roles if possible.														
article	- Use for self-contained content that could be syndicated or stand alone (e.g., a blog post, news article). &lt;br> - May have an associated aria-label or aria-labelledby if it doesn't have a clear heading.														
banner	- Should be used for the primary page header content, typically found at the top of the page. &lt;br> - Usually contains the site name or logo. &lt;br> - Only one banner landmark should be used per document.														
button	- Must be interactive and trigger an action when activated (e.g., via mouse click or keyboard press - Enter or Space). &lt;br> - Should have an accessible name, either through its text content or via aria-label or aria-labelledby. &lt;br> - aria-disabled (state): Indicates that the button is perceivable but not currently operable. &lt;br> - aria-expanded (state): Indicates if the button controls the expansion or collapse of another grouping element. &lt;br> - aria-pressed (state): Indicates if the button is currently pressed (for toggle buttons).														
caption	- Used to provide a descriptive title for a table or grid. &lt;br> - Should be referenced by the aria-labelledby attribute on the table or grid.														
cell	- A cell within a row in a table or grid. &lt;br> - May have aria-colindex and aria-rowindex to explicitly define its position within the table/grid if dynamic or complex. &lt;br> - If it's a header for a row or column, use rowheader or columnheader instead.														
checkbox	- Represents a checkable input that has three possible values: true, false, or mixed. &lt;br> - Must have an accessible name, usually provided by a <label> element or aria-labelledby. &lt;br> - aria-checked (state): Indicates the current checked state. &lt;br> - aria-readonly (state): Indicates that the user cannot modify the value of the checkbox. &lt;br> - aria-required (property): Indicates that the user must check the checkbox before submitting a form.														
columnheader	- A header cell for a column in a table or grid. &lt;br> - Should be within a row with the role="row" attribute. &lt;br> - Use scope="col" on native <th> elements instead when possible. &lt;br> - aria-sort (property): Indicates if and how the column is sorted.														
combobox	- Represents a selectable control that allows the user to choose one or more items from a dynamically displayed list of choices. &lt;br> - Must be associated with the controlled element (e.g., listbox, grid) using aria-controls. &lt;br> - Should have an accessible name, usually via a <label> or aria-labelledby. &lt;br> - aria-expanded (state): Indicates whether the associated popup is currently visible. &lt;br> - aria-autocomplete (property): Suggests what values might be entered by the user. &lt;br> - aria-activedescendant (property): Identifies the currently active descendant of the combobox.														
complementary	- Used for supporting content that is related to the main content but not essential (e.g., a sidebar, related links). &lt;br> - May have an aria-label or aria-labelledby to identify its purpose.														
contentinfo	- Should be used for the page footer content, typically found at the bottom of the page. &lt;br> - Often contains copyright information, links to privacy policies, etc. &lt;br> - Only one contentinfo landmark should be used per document.														
definition	- Provides a definition for a term. &lt;br> - Should be associated with the term it defines, often implicitly within a <dl> list. &lt;br> - May use aria-labelledby to explicitly associate with the term.														
dialog	- Represents an application window that is designed to interrupt the current processing of an application in order to prompt the user to enter information or require a response. &lt;br> - Should have an accessible name, typically via aria-labelledby referencing a heading within the dialog. &lt;br> - Focus management is crucial; focus should be set to the first interactive element within the dialog when it opens and should be trapped within the dialog until it is closed. &lt;br> - aria-modal (state): Indicates that elements outside the dialog are inert.														
directory	- Represents a list of references to members of a group, such as a table of contents. &lt;br> - Should be used for listing items, where each item might link to another part of the content. &lt;br> - Use <ul> or <ol> with appropriate list item semantics where possible.														
document	- The main content of a web page. &lt;br> - The <body> element usually has an implicit document role. &lt;br> - Use role="document" on an <iframe> element to indicate embedded content as a separate document.														
feed	- Represents a dynamic container where content is added and removed over time, similar to a live blog or social media feed. &lt;br> - Should use aria-live to indicate the dynamic nature of the content. &lt;br> - Individual items within the feed should have appropriate roles (e.g., article).														
figure	- Represents content that is often referenced as a single unit, such as an image with a caption, an illustration, a diagram, or a code snippet. &lt;br> - May have an associated figcaption element (which provides the accessible name) or use aria-labelledby or aria-label.														
form	- Represents a section of a document containing interactive controls for submitting information. &lt;br> - Should have an accessible name, either through a <label> associated with the first control or using aria-labelledby or aria-label.														
grid	- Represents an interactive grid containing rows and columns of cells, similar to a table but with interactive elements. &lt;br> - Should have an accessible name, typically via aria-labelledby referencing a related heading or using aria-label. &lt;br> - Requires robust keyboard navigation to move between cells. &lt;br> - May use aria-rowindex, aria-colindex, aria-rowcount, aria-colcount, aria-activedescendant, aria-selected, aria-readonly.														
gridcell	- A cell within a row of an interactive grid. &lt;br> - Must be focusable if it contains interactive content or if the cell itself is interactive. &lt;br> - May have aria-selected if the grid supports selection.														
group	- Used to semantically group related elements. &lt;br> - May have an aria-label or aria-labelledby to provide a label for the group. &lt;br> - Does not inherently affect focus or interaction.														
heading	- Represents a section heading. Use <h1> to <h6> elements for native heading semantics whenever possible. &lt;br> - aria-level (property): Indicates the hierarchical level of the heading if the native heading elements are not used.														
img	- Represents an image. Use the <img> element with an alt attribute to provide an accessible name whenever possible. &lt;br> - If using role="img" on a non-<img> element, provide an accessible name using aria-label or aria-labelledby. &lt;br> - aria-describedby (property): Can be used to provide a longer description of the image.														
link	- Represents a hyperlink. Use the <a> element with an href attribute for native link semantics whenever possible. &lt;br> - Should have clear and concise link text that describes the destination. &lt;br> - If using role="link" on a non-<a> element, ensure it is keyboard focusable (using tabindex="0") and has appropriate keyboard event handling (e.g., Enter key to activate). &lt;br> - aria-disabled (state): Indicates that the link is perceivable but not currently operable.														
list	- Represents a list of items. Use <ul> (unordered list) or <ol> (ordered list) elements for native list semantics whenever possible. &lt;br> - If using role="list" on a non-list element, its children should have the listitem role. &lt;br> - May have an aria-label or aria-labelledby to provide a label for the list.														
listbox	- Represents a list of options from which the user may select one or more items. &lt;br> - Should have an accessible name, typically via a <label> or aria-labelledby. &lt;br> - Requires keyboard navigation to move between options. &lt;br> - aria-multiselectable (property): Indicates whether multiple items can be selected. &lt;br> - aria-activedescendant (property): Identifies the currently focused option within the listbox. &lt;br> - aria-selected (state): Indicates which options are currently selected.														
listitem	- An item within a list or listbox. &lt;br> - Should be a child of an element with the list or listbox role.														
log	- A type of live region where new information is added in a meaningful order, and older information may disappear. Examples include chat logs or error logs. &lt;br> - Should have aria-live set to polite or assertive depending on the urgency of the updates. &lt;br> - aria-atomic (state): May be relevant depending on how updates should be presented.														
main	- Identifies the primary content of the document. &lt;br> - Only one main landmark should be used per document.														
marquee	- Represents a scrolling text or image. Avoid using this role as it can be distracting and difficult for some users to follow. Consider alternative ways to present the information. &lt;br> - If used, ensure the content is also available in a non-scrolling format.														
math	- Represents mathematical content. Use MathML for better semantic representation when possible. &lt;br> - Provide an accessible name or description using aria-label or aria-describedby for screen reader users.														
menu	- Represents a set of user commands that are displayed in a menu. &lt;br> - Should have an accessible name, often via aria-labelledby referencing a related heading or label. &lt;br> - Requires specific keyboard navigation (e.g., arrow keys to move between items, Enter to activate). &lt;br> - Its children should have roles like menuitem, menuitemcheckbox, or menuitemradio. &lt;br> - aria-orientation (property): Indicates whether the menu is horizontal or vertical.														
menubar	- A presentation of menu, usually horizontal, that is intended to be always visible. &lt;br> - Inherits rules and expected attributes of menu. &lt;br> - Typically uses horizontal arrow keys for navigation between top-level items and vertical arrow keys for submenus.														
menuitem	- An option within a menu or menubar. &lt;br> - Must be focusable and activate a command when selected (e.g., via Enter key). &lt;br> - Should have an accessible name (the text of the menu item). &lt;br> - aria-disabled (state): Indicates if the menu item is disabled.														
menuitemcheckbox	- A checkable option within a menu or menubar that has two possible values: true or false. &lt;br> - Inherits rules and expected attributes of menuitem. &lt;br> - aria-checked (state): Indicates the current checked state.														
menuitemradio	- A checkable option within a menu or menubar that is part of a group of radio items, only one of which can be checked at a time. &lt;br> - Inherits rules and expected attributes of menuitem. &lt;br> - aria-checked (state): Indicates whether the current item is checked. &lt;br> - Should be contained within a radiogroup.														
navigation	- Represents a section containing navigational elements (e.g., menus, table of contents). &lt;br> - Should have an accessible name using aria-label or aria-labelledby to describe its purpose (e.g., "Main navigation", "Table of contents").														
none / presentation	- Indicates that the element serves only as a container and has no semantic meaning. &lt;br> - Should be used carefully and generally not on interactive elements, as it removes their semantic information. &lt;br> - If an element has interactive children, apply role="none" or role="presentation" to the parent and ensure the interactive children have appropriate roles.														
note	- Represents content that is ancillary to the main content but important enough to be noticed (e.g., footnotes, annotations). &lt;br> - May be associated with the main content using aria-describedby.														
option	- Represents an option within a select element or a listbox. &lt;br> - Should have an accessible name (its text content). &lt;br> - In a listbox, aria-selected (state) indicates if the option is selected.														
progressbar	- Displays the progress of a task that takes a significant amount of time. &lt;br> - Should have an accessible name, often indicating the task being performed. &lt;br> - aria-valuenow (state): The current value of the progress bar. &lt;br> - aria-valuemin (property): The minimum allowed value. &lt;br> - aria-valuemax (property): The maximum allowed value. &lt;br> - aria-valuetext (property): A human-readable text alternative of the current value.														
radio	- A checkable input in a group of radio controls, only one of which can be selected at a time. &lt;br> - Must have an accessible name, usually provided by a <label> or aria-labelledby. &lt;br> - aria-checked (state): Indicates whether the current radio button is selected. &lt;br> - Should be contained within a radiogroup.														
radiogroup	- A set of related radio controls. &lt;br> - Should have an accessible name using aria-label or aria-labelledby to describe the group. &lt;br> - Only one radio button														
region	- A perceivable section of a document, typically with a heading. Use semantic HTML5 sectioning elements (<article>, <aside>, <nav>, <section>) with headings whenever possible. &lt;br> - Should have an accessible name using aria-label or aria-labelledby to describe its purpose if a native heading is not present or sufficient.																								
row	- A row of cells within a table, grid, or treegrid. &lt;br> - Should contain elements with the cell, columnheader, or rowheader roles. &lt;br> - May have aria-rowindex to explicitly define its position within the table/grid if dynamic or complex. &lt;br> - aria-selected (state): Indicates if the row is selected in a grid or treegrid that supports row selection.																								
rowheader	- A header cell for a row in a table or grid. &lt;br> - Should be within a row with the role="row" attribute. &lt;br> - Use scope="row" on native <th> elements instead when possible. &lt;br> - aria-sort (property): Indicates if and how the related column is sorted.																								
scrollbar	- A control that enables users to adjust the viewable content within a scrolling region. &lt;br> - Should be associated with the scrollable region it controls using aria-controls. &lt;br> - aria-orientation (property): Indicates whether the scrollbar is horizontal or vertical. &lt;br> - aria-valuenow (state): The current value of the scrollbar. &lt;br> - aria-valuemin (property): The minimum allowed value. &lt;br> - aria-valuemax (property): The maximum allowed value. &lt;br> - aria-valuetext (property): A human-readable text alternative of the current value.																								
search	- A section containing a search form. &lt;br> - Should have an accessible name using aria-label or aria-labelledby (e.g., "Search").																								
searchbox	- A type of text input control where users can enter search queries. Use the <input type="search"> element for native semantics whenever possible. &lt;br> - Should have an accessible name, usually provided by an associated <label> or aria-labelledby. &lt;br> - May use aria-autocomplete to provide hints about expected input. &lt;br> - May be associated with a listbox of suggestions using aria-controls and aria-activedescendant.																								
separator	- A divider between content regions that is perceivable but does not represent a structural hierarchy. Use <hr> for a thematic break in content when appropriate. &lt;br> - May have an aria-orientation (horizontal or vertical) if the orientation is visually apparent. &lt;br> - Can be interactive, in which case it might have aria-valuenow, aria-valuemin, aria-valuemax, and aria-valuetext if it functions as a slider.																								
slider	- Allows users to select a value from within a given range. Use <input type="range"> for native semantics whenever possible. &lt;br> - Should have an accessible name, usually provided by an associated <label> or aria-labelledby. &lt;br> - aria-valuenow (state): The current value of the slider. &lt;br> - aria-valuemin (property): The minimum allowed value. &lt;br> - aria-valuemax (property): The maximum allowed value. &lt;br> - aria-orientation (property): Indicates whether the slider is horizontal or vertical. &lt;br> - aria-valuetext (property): A human-readable text alternative of the current value.																								
spinbutton	- A single-line input control that allows the user to increment or decrement a numeric value. Use <input type="number"> with appropriate constraints for native semantics whenever possible. &lt;br> - Should have an accessible name, usually provided by an associated <label> or aria-labelledby. &lt;br> - aria-valuenow (state): The current value of the spinbutton. &lt;br> - aria-valuemin (property): The minimum allowed value. &lt;br> - aria-valuemax (property): The maximum allowed value. &lt;br> - aria-valuetext (property): A human-readable text alternative of the current value. &lt;br> - aria-valuestep (property): The allowed value increment.																								
status	- A type of live region whose content is advisory information for the user but is not so important as to justify an alert. &lt;br> - Should have aria-live set to polite. &lt;br> - aria-atomic (state): Indicates whether assistive technologies will present all or only parts of the changed region.																								
summary	- A summary, caption, or legend for the content of the details element. Use the <summary> element for native semantics whenever possible. &lt;br> - Should be the first child of a details element. &lt;br> - Its text content serves as the accessible name for the details element. &lt;br> - aria-expanded (state) on the parent details element indicates whether the content is expanded or collapsed.																								
tab	- A label providing a single user interface control for a tab panel. &lt;br> - Should be contained within a tablist. &lt;br> - Should have an accessible name (its text content). &lt;br> - aria-selected (state): Indicates whether the current tab is selected. &lt;br> - aria-controls (property): Should refer to the associated tabpanel.																								
table	- Represents data arranged in rows and columns. Use the <table> element for native semantics whenever possible. &lt;br> - Should have an accessible name, typically via a <caption> element (referenced by aria-labelledby) or using aria-label. &lt;br> - Its children should have roles like row, and the children of row should have roles like cell, rowheader, or columnheader.																								
tablist	- A list of tab controls. Use <ul> or <ol> with role="tab" on the list items for non-native implementations. &lt;br> - Should have an accessible name using aria-label or aria-labelledby. &lt;br> - Requires specific keyboard navigation (e.g., arrow keys to move between tabs, Enter or Space to select). &lt;br> - aria-orientation (property): Indicates whether the tabs are arranged horizontally or vertically. &lt;br> - aria-activedescendant (property): May be used to manage focus within the tablist.																								
tabpanel	- A container for the resources associated with a tab. &lt;br> - Should be associated with its corresponding tab using aria-labelledby (referencing the tab) and the tab should control it using aria-controls. &lt;br> - Should be hidden when its associated tab is not selected.																								
term	- A term or phrase with a corresponding definition. Typically used within a <dl> list. &lt;br> - May be associated with its definition (definition role) using aria-describedby on the definition and aria-labelledby on the term (though implicit association within <dl> is preferred).																								
textbox	- A single-line or multi-line text input field where the user can enter and edit text. Use <input type="text">, <input type="email">, <textarea>, etc., for native semantics whenever possible. &lt;br> - Should have an accessible name, usually provided by an associated <label> or aria-labelledby. &lt;br> - aria-autocomplete (property): Suggests what values might be entered by the user. &lt;br> - aria-multiline (property): Indicates whether the textbox accepts more than one line of input. &lt;br> - aria-readonly (state): Indicates that the user cannot modify the value of the textbox. &lt;br> - aria-required (property): Indicates that the user must provide a value for the textbox before submitting a form. &lt;br> - aria-disabled (state): Indicates that the textbox is perceivable but not currently operable.																								
toolbar	- A collection of commonly used function buttons or controls. &lt;br> - Should have an accessible name using aria-label or aria-labelledby to describe its purpose (e.g., "Formatting toolbar"). &lt;br> - Typically contains elements with the button, radio, checkbox, combobox, etc., roles. &lt;br> - May have an aria-orientation (horizontal or vertical) if the orientation is visually apparent.																								
tooltip	- A contextual popup that displays a description for an element when the user hovers over, focuses on, or otherwise interacts with it. &lt;br> - Should be associated with the element it describes using aria-describedby. &lt;br> - Should appear and disappear dynamically without requiring user action beyond the triggering interaction. &lt;br> - Should not trap focus.																								
tree	- Represents a hierarchical list. Use <ul> or <ol> with role="treeitem" on the list items and potentially role="group" for nested levels for non-native implementations. &lt;br> - Should have an accessible name using aria-label or aria-labelledby. &lt;br> - Requires specific keyboard navigation (e.g., arrow keys to navigate, Enter to select/expand/collapse). &lt;br> - aria-activedescendant (property): Identifies the currently focused item in the tree.																								
treegrid	- A grid whose rows can be expanded and collapsed like in a tree structure. &lt;br> - Combines the features and requirements of both grid and tree. &lt;br> - Should have an accessible name. &lt;br> - Requires robust keyboard navigation for both grid and tree interactions. &lt;br> - Uses attributes from both grid (e.g., aria-rowindex, aria-colindex, aria-selected) and tree (e.g., aria-expanded, aria-level).																								
treeitem	- An option within a tree or treegrid. &lt;br> - Should be a child of an element with the tree or treegrid role or a group within the tree. &lt;br> - May contain nested treeitem or group elements. &lt;br> - aria-expanded (state): Indicates whether the item's children are expanded or collapsed. &lt;br> - aria-level (property): Indicates the level of the item in the hierarchy. &lt;br> - aria-selected (state): Indicates if the item is selected. &lt;br> - aria-activedescendant (property) on the parent tree or treegrid may point to the currently focused treeitem.																								
window	- Represents a top-level application window. This role is generally used for desktop applications and is less common in web development. For dialogs within web pages, use the dialog role. &lt;br> - Should have an accessible name, typically via aria-labelledby referencing a title bar or heading within the window. &lt;br> - Focus management is critical; focus should be set appropriately when the window is activated. &lt;br> - May have attributes like aria-modal if it prevents interaction with other parts of the application.
